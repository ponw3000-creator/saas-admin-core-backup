# SaaS AI API 接口契约 (V1.0)

> **声明**：本文件定义了本项目前后端通信的唯一合法结构。后端实现必须以此为准，前端调用必须以此为基。

## 1. 统一响应体 (Standard Response)

所有 API 接口返回必须使用以下 JSON 结构，严禁直接返回业务对象。

| 字段名        | 类型     | 必填 | 说明                                                                  |
| :--------- | :----- | :- | :------------------------------------------------------------------ |
| `code`     | String | 是  | 业务状态码：SUCCESS, BIZ\_ERROR, PARAM\_INVALID, AUTH\_DENIED, SYS\_ERROR |
| `message`  | String | 是  | 针对 code 的可读性说明，用于 UI 提示或 AI 报错分析                                    |
| `data`     | Object | 是  | 业务数据对象，脱敏后的结果，无数据则返回 {}                                             |
| `trace_id` | String | 是  | 请求唯一标识，用于全链路追踪排查                                                    |
| `fallback` | String | 否  | 建议话术，后端预设给 AI 参考的兜底文案                                               |

## 2. 错误处理矩阵与 AI 行为指引

AI 必须根据 `code` 执行对应的交互动作，严禁产生幻觉：

| Code               | 含义     | AI 动作                           |
| :----------------- | :----- | :------------------------------ |
| **SUCCESS**        | 成功     | 解析 data 字段，继续执行或向用户确认结果         |
| **BIZ\_ERROR**     | 业务逻辑拦截 | 停止尝试，将 `message` 原样告知用户（如：余额不足） |
| **PARAM\_INVALID** | 参数校验失败 | 停止尝试，检查传参格式或向用户追问缺失信息           |
| **AUTH\_DENIED**   | 权限不足   | 停止尝试，告知用户无权访问，引导联系管理员           |
| **SYS\_ERROR**     | 系统级异常  | 停止尝试，告知用户"服务繁忙"，引导稍后重试          |

## 3. 防护与脱敏规范

1. **输入防御**：后端必须严格校验 DTO 类型，拦截 LLM 幻觉产生的所有未定义字段。
2. **敏感脱敏**：
   - 手机号：`138****5678`
   - 身份证：`4401**********1234`
   - 严禁在 data 中返回明文密码或 Secret。
3. **调用频率**：同一会话内，同一工具工具调用每分钟限制 10 次，防止死循环。

## 4. 业务字典 (Business Enums)

AI 必须遵守以下枚举，严禁自行发明：

- **渠道类型 (ChannelType)**: `WX_MINI`, `APP`, `WEB`, `H5`
- **会话状态 (TicketStatus)**: `WAITING` (排队), `ACTIVE` (接待中), `RESOLVED` (已解决), `CLOSED` (关闭)
- **客服状态 (AgentStatus)**: `ONLINE`, `BUSY`, `OFFLINE`

### 📊 数据架构契约：流水业务数据的快照原则 (Data Snapshotting Principle)

**🎯 适用场景 (Scope)**
在进行后端数据库表结构设计或 API Payload 设计时，针对所有涉及“不可变历史事实”的流水型业务数据（如：`ChatLogs` 聊天会话记录、`Tickets` 售后工单、`Orders` 交易订单、`ActionLogs` 操作日志等）。

**🚨 核心强制法则 (Core Rule)**
处理上述流水型数据时，**必须采用“快照存储（Snapshotting）”模式**。
在记录创建（Insert）的瞬间，必须将当时关联实体的关键业务维度（如：客服当时的姓名、当时所属的分组 ID 与名称、当时的角色等）作为冗余字段，直接持久化保存在该流水表中。

**📝 防御性业务解释 (Defensive Context)**

- **Why (业务背景)**：基础实体数据（如员工改名、调岗、角色降级）是动态可变的，但历史发生的业务动作是绝对不可变的。快照存储切断了“当前状态”对“历史事实”的干扰。
- **What if (灾难预警)**：如果不使用快照存储，当客服“小明”本月从【售前组】调至【售后组】后，系统在拉取上个月的报表时，会因为动态联表查询，将小明在售前组的百单业绩全部错误归入售后组，导致企业数据大盘彻底崩溃。

**❌ 绝对禁令 (Anti-Pattern - 严禁执行)**
在统计、导出或渲染历史报表时，**严禁**仅依靠单一外键（如 `user_id`），去实时联表查询（JOIN）或关联当前的实体表（如 `Users` 或 `Departments` 表）来获取显示名称或归属状态。

**✅ 数据结构设计基准 (Best Practice Model)**
设计此类数据模型时，必须包含 `_at_time` 或类似后缀的快照字段：

```json
// ❌ 错误示范：仅依赖外键（高度耦合当前状态）
{
  "chat_id": 1024,
  "user_id": 8, // 如果 user 8 被物理删除，此条记录将成为孤岛报错
  "content": "您好，很高兴为您服务"
}

// ✅ 正确示范：冗余维度的快照存储（解耦且安全）
{
  "chat_id": 1024,
  "user_id": 8, // 维持外键关联，用于活人数据的聚合
  "user_name_at_time": "浩哥", // 发生时的真实姓名
  "group_id_at_time": 2, // 发生时所在的部门ID
  "group_name_at_time": "大客户服务组", // 发生时所在的部门名称
  "role_at_time": "supervisor", // 发生时的角色权限
  "content": "您好，很高兴为您服务"
}
```
